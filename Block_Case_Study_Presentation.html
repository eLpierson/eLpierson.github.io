Solution

Technical Challenges Overcome

Challenge 1

Settings Trapped in Contracts

Problem

80% of pros called within first week to change service area/budget. Every change required legal contract amendment. 3-day turnaround.

Root Cause

Architecture put ALL pro preferences into legal contract. Settings should have been profile data (adjustable), not contract terms (fixed).

Solution

Created settings taxonomy: Profile settings (mutable) → Salesforce custom objects. Contract terms (immutable) → Ironclad. Contract references profile via Salesforce ID. Aligned with legal counsel that profile changes aren't contract amendments.

Result: Settings changes went from 3-day turnaround to instant. Zero contract re-signs needed.
Challenge 2

Manual Fraud Review Bottleneck

Problem

Fraud team manually reviewed every pro AFTER sales call + onboarding. Daily batch export. Rework loops if declined.

Root Cause

Fraud team had no real-time access to enrollment data, worked from daily batch export. Process ordering was wrong.

Solution

Real-time Socure identity verification integrated during enrollment. Auto-approve low-risk (~60%), auto-reject high (~20%), flag medium for review (~20%).

Result: 60% of pros auto-activated instantly. Flagged reviews dropped from 2 days to 4 hours.
Challenge 3

Map-Based Service Area Selection

Problem

Service area stored as zip codes, but pros think in drive time / radius. UX mismatch with data model.

Solution

Google Maps API with radius drawing tool. Geocoding converts address → lat/long. Backend calculates all zip codes within radius using centroid data. Stores both visual radius (display) and zip list (matching).

Edge Case

Radius crossing state lines = different pricing zones. Built business logic for cross-boundary selection.

Result: Intuitive UX matching pro mental model while maintaining backend data structure for matching algorithm.
Challenge 4

Refund Automation (Reverse Logistics)

Problem

Defective leads (wrong phone, spam) required multi-day manual refund. 40% of care calls = refund requests.

Solution

Self-serve refund workflows according to company policies. Human review if suspicious (>30% dispute rate over 1 year).

Technical

Rules engine evaluates lead quality score + pro dispute history. Auto-approve and auto-process refund if criteria met.

Result: Refund care calls dropped from 40% to 10%. Closed 1 of 3 call centers (120 → 80 agents).

Execution

Phased Rollout: De-Risking Through Partnership

Phase 1 -3 Months

Fix the Foundation

Built infrastructure without changing sales process. Salesforce integration, real-time fraud API, settings decoupling, event architecture. Sales team kept doing phone enrollments - but systems got cleaner underneath.

Why it worked: Sales reps saw immediate benefits (real-time status). Support team saw benefits (no more contract amendments). Didn't threaten anyone's workflow or compensation yet. Built trust.
Phase 2 -2 Months

Limited Cohort Test in 2 Confidential Test Zones

Controlled experiment: organic leads in 2 markets → self-serve flow. Comparable markets as control group. Daily dashboards tracking conversion, CSAT, cost, abandonment.

Results: Conversion +10%, time -98%, cost -87%, CSAT +23pts, abandonment halved. Self-serve was better for customers AND cheaper.
Decision Gates: Established clear kill criteria upfront - would shut down experiment if conversion dropped >10%, CSAT declined, or abandonment increased. Data-driven framework ensured defensible go/no-go decision.
Phase 3 -3 Months

National Rollout (10% → 25% → 50% → 100%)

Gradual scaling with monitoring and hypercare. Sales team transitioned to more intentionally high-touch roles as partners, not victims.

End state: Self-serve default for organic SMB leads. Phone-based retained for outbound enterprise/partnerships (high-touch still valuable there) and inbound SMB.

Experiment Terms We Agreed On with Sales

"Test in 2 markets for 60 days"
"If conversion drops >10%, we kill it immediately"
"Sales reps get CREDIT for self-serve enrollments - comp protected"
"Self-serve for organic, phone for outbound/complex"
"Clear kill criteria - data-driven, defensible decision"

Learning + Closing

What I'd Do Differently & Why It Matters for Block

What I Did

Identified pro enrollment (B2C) as the constraint blocking Jobs Done Well. Fixed it comprehensively - 7+ days down to 12 minutes, 87% cost reduction, national scale.

What I Missed

I solved the loudest constraint instead of mapping the constraint tree. Partner activation (B2B) was equally constraining supply growth - onboarding also took 10+ days - and I didn't see it until after enrollment was fully shipped. I scoped one node instead of the system.

What It Cost

Months of delayed geographic expansion into underserved markets. More Jobs Done Well left on the table. A slower path to positive unit economics. The partner fix was smaller scope - if I'd resourced it in parallel, we'd have captured that supply growth much sooner.

How I Think About Product Strategy Now
Before: Solve the biggest problem → celebrate.
After: Map the constraint tree → fix prerequisites → parallelize → measure the new constraint → iterate. Always ask: "If we fix THIS, what breaks next?"
